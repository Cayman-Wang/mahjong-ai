## Findings

- 标题: 近期动作历史包含换三张选牌，导致信息泄露
  - 严重级别: P1
  - 位置: [src/mahjong_ai/core/engine.py](src/mahjong_ai/core/engine.py#L314-L323)、[src/mahjong_ai/env/obs_vector_encoder.py](src/mahjong_ai/env/obs_vector_encoder.py#L203-L255)
  - 现象/风险: `recent_actions` 记录了 SWAP_PICK 的 `action_id`，而观测编码会将 `action_id` 归一化写入所有玩家的近期动作历史特征，等价于把对手在换三张时选的隐藏牌暴露给所有玩家。训练会在信息泄露环境下收敛，评测失真。
  - 复现步骤:
    1. 使用默认规则（swap_enabled=True）。
    2. 在 `SWAP_PICK_1` 阶段执行 `engine.step()`。
    3. 调用 `encode_observation_vector(state, pid=1)`，检查 `RECENT_ACTION_SLICE`，可解码出其他玩家的 SWAP_PICK 选牌。
  - 根因: 换三张的选牌属于隐藏信息，但 [src/mahjong_ai/core/engine.py](src/mahjong_ai/core/engine.py#L314-L323) 仍通过 `_record_recent_action` 写入 `ActionTrace`，而 [src/mahjong_ai/env/obs_vector_encoder.py](src/mahjong_ai/env/obs_vector_encoder.py#L203-L255) 无条件编码 `recent_actions`。
  - 最小修复建议:
    ```python
    # 方案 A：记录时屏蔽隐藏动作
    if da.kind in {ActionKind.SWAP_PICK}:
        self._record_recent_action(state, pid=pid, action_id=encode_action(ActionKind.PASS))
    else:
        self._record_recent_action(state, pid=pid, action_id=actions[pid])
    ```
    或在编码侧过滤：对 `SWAP_PICK` 一律写入 padding slot，不写 action_id。
  - 回归测试建议:
    - 新增 `test_recent_action_history_does_not_leak_swap_pick`：在 swap 阶段后编码对手观测，断言 recent_actions 中不存在 `SWAP_PICK` 的实际 tile id。

- 标题: 告警静音默认行为不符合“可配置静音”的要求
  - 严重级别: P2
  - 位置: [src/mahjong_ai/training/rllib_runner.py](src/mahjong_ai/training/rllib_runner.py#L268-L338)
  - 现象/风险: `_build_algorithm()` 无条件调用 `_install_ray_warning_filters()`，直接忽略 Ray 的 DeprecationWarning 和 RayDeprecationWarning。即使未设置 `warnings.quiet_*`，也会静默吞掉部分告警，违背“默认保留、可配置静音”的要求。
  - 复现步骤:
    1. 运行训练入口且不设置 `warnings.quiet_*`。
    2. 观察 Ray DeprecationWarning 不再输出（被 `_install_ray_warning_filters()` 全局过滤）。
  - 根因: `_install_ray_warning_filters()` 未受配置控制，且在 `_build_algorithm()` 中无条件执行。
  - 最小修复建议:
    ```python
    # 仅在显式开启时静音
    if quiet_ray_future_warning or quiet_new_api_stack_warning:
        _install_ray_warning_filters()
    ```
  - 回归测试建议:
    - 新增 `test_warning_filters_respect_quiet_flags`：quiet=false 时不应屏蔽 DeprecationWarning。

- 标题: illegal_action_rate 指标在出现非法动作时不可用
  - 严重级别: P2
  - 位置: [src/mahjong_ai/training/rllib_runner.py](src/mahjong_ai/training/rllib_runner.py#L601-L686)
  - 现象/风险: 一旦检测到非法动作，函数直接抛 `RuntimeError`，导致 `illegal_action_rate` 永远无法非零输出（有问题时直接中断训练/评测）。指标本身无法量化非法动作风险。
  - 复现步骤:
    1. 强行构造一个不使用 action mask 的策略（或临时在 `_compute_policy_action()` 中返回非法动作）。
    2. 运行评测，直接抛错，`illegal_action_rate` 永远无法被记录。
  - 根因: 评测逻辑将非法动作作为 fatal error，而不是统计指标。
  - 最小修复建议:
    ```python
    if illegal:
        illegal_attempts += 1
        aid = first_legal(mask)  # 或 continue 并记录非法
    ```
  - 回归测试建议:
    - 新增 `test_illegal_action_rate_counts_errors`：构造非法动作路径，断言 `illegal_action_rate > 0` 且不抛异常。

## 重点风险清单回答（是/否 + 证据）

1. 新增观测特征是否造成信息泄露？**是**。SWAP_PICK 的 action_id 被编码进 recent_actions（见 [engine.py](src/mahjong_ai/core/engine.py#L314-L323) + [obs_vector_encoder.py](src/mahjong_ai/env/obs_vector_encoder.py#L203-L255)）。
2. 是否存在观测维度或切片错位导致训练 silently wrong？**否**。编码末尾做长度断言（见 [obs_vector_encoder.py](src/mahjong_ai/env/obs_vector_encoder.py#L256-L257)），并有单测覆盖。
3. action mask 是否在所有 forward 路径都生效？**是**。`_forward_inference/_forward_exploration/_forward_train` 均调用 `_mask_action_logits`（见 [rllib_action_mask_rl_module.py](src/mahjong_ai/training/rllib_action_mask_rl_module.py#L123-L151)）。
4. 是否存在非法动作被采样的可能路径？**否（按当前 mask 逻辑）**。mask 覆盖所有 forward 路径，且评测端对非法动作直接抛错（见 [rllib_runner.py](src/mahjong_ai/training/rllib_runner.py#L653-L656)）。
5. opponent pool 机制是否真的参与了对局？**是**。`_build_self_play_policy_ids` 创建对手池，`policy_mapping_fn` 按 seat/episode 选择主策略或对手，且每 `snapshot_interval` 同步权重（见 [rllib_runner.py](src/mahjong_ai/training/rllib_runner.py#L820-L929)）。
6. 评测指标（尤其 illegal_action_rate）是否可信且可复现？**否**。非法动作会抛错中断，无法产生非零指标（见 [rllib_runner.py](src/mahjong_ai/training/rllib_runner.py#L653-L686)）。
7. warning 静音开关是否可控且默认行为正确？**否**。默认会调用 `_install_ray_warning_filters()` 静默部分告警（见 [rllib_runner.py](src/mahjong_ai/training/rllib_runner.py#L268-L338)）。
8. 是否还有未清理的旧 API/旧实现残留会导致升级风险？**否（误报）**。旧 ModelV2 文件已移除。

## 2) Findings（按严重度）

- 标题: `PENG/GANG_MING` 未回收点炮牌，导致牌守恒与可见牌统计错误
- 严重级别: P0
- 位置: `src/mahjong_ai/core/engine.py:607`、`src/mahjong_ai/core/engine.py:638`、`src/mahjong_ai/core/engine.py:475`
- 现象/风险:
  - 出牌时已执行 `p.discards.append(tid)`（`engine.py:475`）。
  - 响应 `GANG_MING`/`PENG` 时，只从手牌扣 3/2 张并加入 meld，但未将被碰/明杠那张从 `discarder.discards` 移除。
  - 结果：总牌数 +1；同一 tile 在 `discards + melds` 可能出现 5 次（>4），破坏零和与观测一致性。
- 复现步骤:
  1. 构造 `RESPONSE` 阶段：A 刚打出 tile `t`，B 可 `PENG` 或 `GANG_MING`。
  2. 执行 `engine.step(state, {B: encode_action(PENG/GANG_MING), others: PASS})`。
  3. 统计 `hands + discards + melds + wall`，可见 `after - before = +1`。
- 根因:
  - 状态机把“被副露拿走的弃牌”保留在弃牌堆，未做一次性转移。
- 最小修复建议:
  - 在 `PENG` 与 `GANG_MING` 成功分支前，校验并弹出 `state.players[discarder].discards[-1] == t`，再进行 `meld` 结算。
  - 伪代码：
    ```python
    last = state.players[discarder].discards
    if not last or last[-1] != t:
        raise ValueError("discard stack mismatch")
    last.pop()
    ```
- 回归测试建议:
  - `test_peng_keeps_total_tiles_constant`
  - `test_ming_gang_keeps_total_tiles_constant`
  - 断言：全局牌计数守恒、任意 tile 公开可见计数 `<= 4`。

- 标题: `recent_actions` 暴露 `SWAP_PICK(tile)`，导致信息泄露
- 严重级别: P0
- 位置: `src/mahjong_ai/core/engine.py:322`、`src/mahjong_ai/env/obs_vector_encoder.py:203`
- 现象/风险:
  - 换三张阶段把 `SWAP_PICK(tile)` 真实 `action_id` 写入 `recent_actions`。
  - 观测编码将其无差别编码到所有玩家向量中，非当前玩家可恢复对手私有选牌。
  - 自对弈会在“泄露环境”学习，评测失真，线上规则等价性失效。
- 复现步骤:
  1. `seed=2` 开局，进入 `SWAP_PICK_1`。
  2. 执行一次 `step` 后读取玩家 1 的观测向量 `RECENT_ACTION` 片段。
  3. 可解码出玩家 0 的 `swap_pick_tid`；同时玩家 1 手牌该 tile 计数可为 0。
- 根因:
  - `recent_actions` 被当作“公共历史”，但其中混入了本应私有的换牌细节。
- 最小修复建议:
  - 首选：`SWAP_PICK_*` 阶段不写入 `recent_actions`。
  - 备选：编码器遇到 `SWAP_PICK` 仅保留动作类型，不保留 `arg/tile`。
- 回归测试建议:
  - `test_swap_pick_tile_not_visible_to_other_players`
  - 断言：对手观测中不可恢复 `SWAP_PICK` 的 tile id。

- 标题: `_build_algorithm()` 异常时未进入清理域，Ray 资源泄漏
- 严重级别: P1
- 位置: `src/mahjong_ai/training/rllib_runner.py:891`、`src/mahjong_ai/training/rllib_runner.py:976`
- 现象/风险:
  - `algo = _build_algorithm(algo_config)` 在 `try/finally` 之外。
  - 若构建阶段抛错，`algo.stop()` 与 `ray.shutdown()` 都不执行。
  - 多次失败重试会留下初始化状态或后台资源，影响后续训练稳定性。
- 复现步骤:
  1. mock `_build_algorithm` 抛 `RuntimeError("boom")`。
  2. 调用 `train_with_rllib()`。
  3. 检查 `ray.shutdown` 未被调用。
- 根因:
  - 清理逻辑覆盖了“构建后”异常，但未覆盖“构建中”异常。
- 最小修复建议:
  - 将 `_build_algorithm` 放入同一 `try` 块；`algo` 先置 `None`。
  - `finally` 中 `if algo is not None: safe_stop(algo)`，随后无条件（按 started_ray）`ray.shutdown()`。
- 回归测试建议:
  - `test_train_with_rllib_build_failure_still_shuts_down_ray`。

- 标题: `illegal_action_rate` 指标在异常路径不可用
- 严重级别: P2
- 位置: `src/mahjong_ai/training/rllib_runner.py:653`
- 现象/风险:
  - 评测遇到非法动作即 `raise RuntimeError`，训练/评测提前中断。
  - `illegal_action_rate` 设计存在，但难以形成连续监控序列。
- 复现步骤:
  1. 让策略返回非法动作（例如 mock `_compute_policy_action` 返回越界索引）。
  2. 运行评测函数。
  3. 结果直接抛错，无法输出包含非零 `illegal_action_rate` 的报告。
- 根因:
  - 指标统计与 fail-fast 混在同一分支，异常优先级覆盖了可观测性。
- 最小修复建议:
  - 增加配置开关：`strict_illegal_action=true` 时保持抛错；否则记录并回退到首个合法动作继续对局。
- 回归测试建议:
  - `test_illegal_action_rate_counts_when_not_strict`。
